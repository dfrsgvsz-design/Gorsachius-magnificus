## 1. 背景：为什么会出现 217 vs 223

`backend_run.log` 启动期的 WARNING：

```
2026-04-26 12:11:01 [WARNING] field_survey_platform:
  Species mapping (223) does not match checkpoint head (217);
  trimming runtime mapping to checkpoint classes.
2026-04-26 12:11:01 [INFO] field_survey_platform: Loaded model: v4-student, 217 species, val_acc=0.6167706167706167
```

**根因（基于实际仓库证据，由 `scripts/algo_d/audit_species_head_gap.py` 复现）：**

| 项 | 值 | 文件 |
|---|---|---|
| 训练 manifest（GROUND TRUTH） | 19401 条记录、223 unique species | `species_monitoring_platform/data/xc_expanded/manifest.json` |
| 运行时 mapping | 223 类 | `species_monitoring_platform/backend/checkpoints/species_mapping.json` |
| 当前 checkpoint head | **217** | `species_monitoring_platform/backend/checkpoints/best_model.pth`（v4-student, SE-ResNet18） |
| 当前 calibration | T=0.665, ECE_after=0.0432, model=v3-ensemble | 同目录 `calibration.json` |
| 触发点 | `_align_species_mapping_to_checkpoint()` 静默 trim 末尾 6 类 | `backend/main.py:724-748` |

**被静默丢弃的 6 个物种**（生产推理永远不会输出，但 UI/导出可能仍然让用户以为系统在识别）：

| idx | 学名 | 中文名（来自 species DB） | manifest 记录数 |
|----:|------|--------------------------|---------------:|
| 217 | _Upupa epops_           | 戴胜       | 100 |
| 218 | _Urocissa erythroryncha_| 红嘴蓝鹊   | 100 |
| 219 | _Vanellus cinereus_     | 灰头麦鸡   |  94 |
| 220 | _Vanellus vanellus_     | 凤头麦鸡   | 100 |
| 221 | _Zoothera dauma_        | 虎斑地鸫   | **18** ← 风险类 |
| 222 | _Zosterops japonicus_   | 暗绿绣眼   | 100 |

> 这条 trim 行为是上线一个**功能-命名不符**的灰区：物种 DB 写了 223 种，模型实际只能给 217 种打分，剩下 6 种走 fallback 或者直接返回 zero confidence。Google Play 审核策略 ([App Store metadata accuracy](https://play.google.com/about/restricted-content/inappropriate-content/)) 明确禁止"功能与描述不符"，这是工单 P0 的来源。

## 2. 三个候选方案与裁定

| 维度 | Option 1 训练侧 | Option 2 数据侧（mapping 锁到 217） | Option 3 混合（先 2 后 1） |
|---|---|---|---|
| 6 类是否被主模型识别 | 是 | 否（走 BirdNET 降级 / 标灰） | 短期否 / 长期是 |
| 工时 | 数据预热 0.5 d + GPU 训练 4–8 h + 校验 0.5 d | 0.5 d 脚本 + UI/导出同步调整 | 短期 + 长期叠加 |
| 风险点 | _Zoothera dauma_ 仅 18 条 → 该类 val_acc 偏低 | 6 类永久"主模型不可识别"，需要前端 UI 显式声明 | 短期窗口期 mapping 仍非真相 |
| 与 v7 训练管线契合度 | 完全契合（`train_gpu_v7.py` 已是 manifest 驱动） | 反向打补丁，长期欠债 | 短期欠债 |
| Google Play 描述一致性 | ✅ 严格匹配 | ⚠️ 需要文案 + UI 双向声明，否则仍有风险 | ⚠️ 同 Option 2 |

**裁定：Option 1。** 理由：
1. `train_gpu_v7.py` 本来就读 `manifest.json` 推导 `num_species`，不存在"训练管线无法支持 223"的技术阻碍；
2. 5/6 物种样本量 ≥ 94，能上；剩 1 个（_Zoothera dauma_, 18 条）按"低样本类别"处置（见 §4 风险缓解）；
3. Option 2 把"模型 = 物种监测 App"这个根本契约写薄，长期负债更高，与产品定位矛盾。

## 4. 风险与缓解

| # | 风险 | 概率 | 缓解 |
|---|------|-----:|------|
| 1 | _Zoothera dauma_ 仅 18 条 → 该类 recall/precision 显著低于其他 | 高 | **已决策走数据补采**。首选通道 A（iNat 同义名 + 许可证过滤，已实测可拿 67 条）；XC/Macaulay 作备份。详见 §3.0 |
| 1b | 物种 _Zoothera dauma_ 在 IOC v14 已被拆成 4 个种，外部源 API 不再用 lumped 名 | 高 | iNat 通道用 `--also-as` 把 `Z. aurea / Z. major / Z. neilgherriensis` 都查一遍，统一回写为 `Zoothera dauma` 这个 legacy 标签（保 species_mapping.json 不动）。**长期看应在 W3 把 mapping 跟进 IOC v14 拆分**——记为新工单 |
| 2 | v7 ConvNeXt 显存占用比 v4 SE-ResNet 高 → 训练机 OOM | 中 | `train_gpu_v7.py` 已有 `gradient_checkpointing=True`、`accumulate_steps=4`、`batch_size=16`，从最小可用起步；如仍 OOM 把 `--batch-size 8 --accumulate 8` |
| 3 | 整体 val_acc 比 v4-student 0.617 显著下滑（>5%） | 中 | 训练完进入"门禁阶段"：用 `_check_acc_regression.py`（见 §5）对比；若回归则回滚到 v4 + 立刻进 Option 2 |
| 4 | OOD 阈值标定 + 温度标定语义混淆 | 中 | §3.3 明确分两步：先 V7 内置 OOD calibrate（产物在 .pth 内），再独立温度标定（产物在 calibration.json） |
| 5 | 部署切换非原子，期间用户请求出错 | 低 | 用 §3.4 的 `Copy-Item -Force` 一次性切；建议在维护窗口执行（后端 uvicorn 进程重启时长 < 30s） |

## 6. 与 P0 W1 另一条（外化 T/ECE 到 platform_config.json）的接口

- 本决策只负责把 `calibration.json` 的 `temperature/ece_before/ece_after/model_version` **写正确**；
- "外化到 `platform_config.json`" 是 **A 工程师** 的职责，但需要算法 D 给一个稳定 JSON Schema。Schema 提议：

```json
{
  "model_calibration": {
    "model_version": "v7-student",
    "temperature": 0.665,
    "ece_before": 0.2501,
    "ece_after":  0.0432,
    "accuracy":   0.6216,
    "frozen_at":  "2026-06-10T08:00:00Z",
    "source_calibration_json": "backend/checkpoints/calibration.json"
  }
}
```

后端启动时 A 负责把 `backend/checkpoints/calibration.json` 内容**复制**到 `platform_config.json.model_calibration` 字段（避免双源真相），并在 `/api/config` 端点暴露。前端取 `model_calibration.temperature` 做 softmax 缩放。**算法 D 只承诺 `calibration.json` 是 source of truth**。
