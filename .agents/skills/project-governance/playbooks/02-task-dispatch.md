# Playbook 02 · 团队设计与工单下发

## A · 团队设计原则

### 1. DRI 单点负责

每个角色必须有**且只有 1 个**人。如果一个角色无人，PM 临时挂名并立刻招/借/转人。

### 2. 4 ± 1 是甜蜜区

| 团队规模 | 配置 |
|---|---|
| 1 人（单兵）| PM 也是 ta；本制度只是 ta 给自己用 |
| 2–3 人 | PM + 全栈 + UI/运营兼职 |
| **4 人（推荐）** | A 后端 + B 前端 + C QA + D 算法 |
| 5 人（推荐+1）| 加 E UI/运营 |
| 6–8 人 | 在 B 下加 Android 专员 + iOS 专员；在 D 下加数据工程师 |
| 9+ | 拆 sub-PM，本制度复制为多个小组 |

### 3. 反"专家依赖"

- 任何"只有 A 会"的领域，3 个月内 PM 必须促成"另一人能改"
- 任何文档级知识必须落 git，不能只在某人脑子里
- 任何账号/密钥必须 vault 化 + 双人可访问

## B · 工单的"7 字段"硬要求

```
工单 #<工号> — <角色名>

【思路定调】 1–3 句话，告诉 ta 在项目中的本质职责 + 思维框架
【任务清单】 按 P0/P1/P2 + 按周分桶
【验收标准】 可量化、可复现、可双签
【DRI】     单点负责人（一般就是收单人本身）
【截止】    精确到日，最好精确到时
【依赖谁】  上游、需要谁先交付什么
【谁依赖你】下游、谁等你交付什么
```

**缺 1 字段 = 工单作废，返工重写**。

## C · 思路定调的写法（重要）

"思路定调"是 PM 工单里**唯一不可机械化**的部分。它决定 ta 接到工单后能不能进入正确心态。

写作公式：

```
你不是 X，你是 Y。<可量化的本质职责>。
<不做就会发生的具体坏事>。
```

### 好例子

> **A · 后端**：你不是"功能开发"，你是**让后端从 demo 跨进生产的人**。后端只要还在 demo 模式，前端再花哨也只是演示页。

> **B · 前端**：你不是"画 UI 的"，你是**用户摸到 App 时唯一的代理人**。审核员看到的、用户截图发微博的，都是你的产物。

> **C · QA**：你不是"测试员"，你是**上线流水线的安全员**。所有人的代码都要经过你，包括 PM。

> **D · 算法**：你的工作决定"这 App 到底是不是物种监测 App"。任何 mapping 不一致 = 审核员说"功能名不副实"。

> **E · UI/运营**：技术再 perfect，少一张 1024×500 也照样上不了 Play。你是**把项目送过审的最后一道工序**。

### 烂例子（不要写）

- "你要做好后端工作" —— 太空
- "请重视代码质量" —— 套话
- "我们对你有很高期望" —— 鸡汤
- "把所有任务高质量完成" —— 没信息量

## D · 任务清单的"按优先级分桶"

固定格式：

```markdown
1. 【P0 W<n>】<动词>……<改的文件路径 / 模块>……<出口标准>
2. 【P0 W<n>】……
3. 【P1 W<n>】……
4. 【P2 持续】……
```

**禁用**：
- "继续推进 X" → 改成"把 X 从 70% 推到 100%"
- "支持 Y" → 改成"为 Y 提供 <具体接口>"
- "完善 Z" → 改成"修复 Z 的 <具体 bug 编号>"

**P0 / P1 / P2 比例建议**：3 / 4 / 2（一周 9 件事，前 3 件保命）

## E · 验收标准的"三可"

| 标准 | 解释 | 反例 |
|---|---|---|
| **可量化** | 数字、Y/N、PASS/FAIL | "运行良好" ❌ |
| **可复现** | 给出命令/脚本/查询路径 | "测试通过" ❌ |
| **可双签** | PM + 同组任一队友可独立验证 | "本人确认" ❌ |

### 好例子

```
验收标准:
- `curl https://<prod>/api/health` 24h 内连续 99% 返回 200（监控截图为证）
- `release_gate.ps1` ALL PASS（CI 链接）
- `taxonomy/search` 0 个 422（grep 最近 48h 日志）
- 双签: PM + C 工号
```

### 烂例子

```
验收标准:
- 后端稳定运行 ❌
- 接口工作正常 ❌
- 单元测试通过 ❌
```

## F · 工单完成度跟踪

PM 每周一拿出 5 份工单逐条核对：

```markdown
| 工号 | 上周承诺 P0 | 完成 X/Y | 漏的原因 | 本周 P0 调整 |
|---|---|---|---|---|
| A | 3 | 2/3 | 第 3 项被 D 阻塞 | 补做 + 升级 D |
| B | 2 | 2/2 | - | 新 2 项 |
| C | 3 | 1/3 | C 病假 1 天 + Y 项重写 | 缩到 2 项 |
| D | 2 | 0/2 | 决策卡 sponsor | escalate D-04 |
| E | 4 | 4/4 | - | 新 3 项 |
```

完成率连续 2 周低于 60% 的工号：进 1on1 主题排查（不是惩罚，是排原因）。

## G · 工单生命周期

```
DRAFT (PM 草稿)
   │
   ├──► PUBLISHED (周一启动会发出)
   │       │
   │       ├──► IN-PROGRESS (DRI 接单)
   │       │       │
   │       │       ├──► BLOCKED (升级)
   │       │       │       │
   │       │       │       └──► RESUMED
   │       │       │
   │       │       └──► DONE (双签)
   │       │
   │       └──► RE-ESTIMATED (PM 与 DRI 周会调整范围)
   │
   └──► CANCELLED (项目方向变 / 降优先级)
```

任何状态变更必须在工单文件顶部加一行：

```markdown
- 2026-05-08 09:30 IN-PROGRESS → BLOCKED · 原因：等 sponsor 决策 D-04
- 2026-05-09 14:00 BLOCKED → RESUMED · D-04 已决断
- 2026-05-10 18:00 RESUMED → DONE · 双签 PM + C
```

## H · 跨职能依赖矩阵（PM 必维护）

每周更新一次，挂在 governance/ 下：

```markdown
| 上游 | 交付物 | 下游 | 截止 | 状态 |
|---|---|---|---|---|
| A | taxonomy API 契约定稿 | B, D | W1 周三 | 🟢 |
| D | species mapping 决断 | A | W1 周五 | 🟡 |
| C | requirements-dev.txt | A | W1 周一 | 🟢 |
| E | 隐私 URL 公网部署 | (所有人) | W1 周五 | 🔴 |
| ... | ... | ... | ... | ... |
```

任何 🔴 进 RAID Log，任何 🟡 进周会议程。

## 红线

- **PM 不写 ta 不懂的工单** —— 不懂就先和 DRI 聊 30 分钟再写
- **PM 不修改已下发工单的 P0** —— 要改走 D-NN 决策流程
- **DRI 接单后 24h 内必须有书面反馈** —— 默认接受 or 提出异议
- **没双签的"DONE"不算 DONE**
