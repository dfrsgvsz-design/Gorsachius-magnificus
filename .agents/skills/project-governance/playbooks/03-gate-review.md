# Playbook 03 · 闸门评审与 Go/No-Go 决策

## A · 闸门的本质

闸门**不是**项目验收，闸门是 PM 与 sponsor 之间的"信用结算点"——把过去一个周期的"承诺"与"事实"对账，决定下个周期的方向。

> 一句话：**闸门让团队失败得早、便宜、可逆。**

## B · 闸门频次（按项目阶段）

| 阶段 | 频次 | 例子（本项目）|
|---|---|---|
| 红线收敛期 | 周末 1 次 | Gate W1 / W2 / W3 / W4 |
| 平稳推进期 | 双周 1 次 | 上线后 8 周内 |
| 长线维护期 | 月度 1 次 | 1.x 维护版本 |
| 重大变更点 | 即时 | 换主体 / 换栈 / 换 sponsor |

## C · 闸门会议前的准备（PM 一周提前布）

| 时间 | 谁 | 做什么 |
|---|---|---|
| T-7 天 | PM | 给 5 个 DRI 发"闸门预通知"，附 9 项打分维度 |
| T-3 天 | 每 DRI | 提交本维度证据初稿（截图 / 日志 / 链接）|
| T-2 天 | PM | 闭门预演——把 9 项灯先打一次，识别"危险灯"|
| T-1 天 | PM | 给危险灯 DRI 单独 1on1，确认能否赶在会上变绿 |
| T-0 | 全员 | 会议（30–60 min）|

## D · 闸门会议议程（固定 4 段）

```
Part 1 (5 min)  · PM 重申本次闸门"过 / 不过"的判据（9 项打分规则）
Part 2 (20 min) · 5 个 DRI 各 4 min：本维度灯色 + 关键证据 + 风险
Part 3 (15 min) · PM 主持打分 → 输出 R/Y/G 总表
Part 4 (10 min) · 共同决断：GO / CONDITIONAL GO / NO-GO + 补救计划
```

**任何"我去查一下再回复"** = PM 当场降为 🔴。证据不在场 = 状态不存在。

## E · 9 项打分维度（本项目）

| # | 维度 | 红/黄/绿判据 |
|---|---|---|
| 1 | P0 缺陷数 = 0 | 🟢=0 / 🟡=1–2 且有 24h 内修复 PR / 🔴=≥3 或无修复路径 |
| 2 | release_gate ALL PASS | 🟢=全绿 / 🟡=1 项黄 / 🔴=任意 FAIL |
| 3 | E2E 核心路径 | 🟢=全绿 / 🟡=断 1 步 / 🔴=断 ≥2 步 |
| 4 | 生产 24h 99% 200 | 🟢=≥99% / 🟡=95–98.9% / 🔴=<95% |
| 5 | 签名/口令 vault 化 | 🟢=磁盘 0 残留 / 🟡=残留但本周清 / 🔴=未启动 |
| 6 | 商店素材 | 🟢=100% / 🟡=80–99% / 🔴=<80% |
| 7 | 审核账号 24h 可达 | 🟢=测试通过 / 🟡=有但未测 / 🔴=未发 |
| 8 | 回滚演练 | 🟢=演练过 / 🟡=有 SOP 未演 / 🔴=无 SOP |
| 9 | 崩溃监控 + mapping | 🟢=闭环 / 🟡=部分 / 🔴=未启用 |

## F · 决策输出（三选一）

### GO

- 触发：9 项全 🟢
- 含义：进入下一阶段，无附加条件
- 动作：PM 在 Decision Log 写 D-NN「Gate W<n> GO」+ 双签 + 解锁下个 sprint

### CONDITIONAL GO

- 触发：9 项有 🟢，≤2 项 🟡，0 项 🔴，且 🟡 有明确补救
- 含义：继续推进，但带"补救清单"
- 动作：
  - 补救清单**会上写完**，不许"会后补"
  - 每条补救必须有 DRI + 截止时间（不超过 1 周）
  - 下次闸门前 PM 单独验收补救清单

### NO-GO

- 触发：任意 1 项 🔴
- 含义：当前阶段不允许进入下一阶段
- 动作：
  - 不是失败！是"少坑了一次用户"
  - PM 当场决断：
    - 选项 1：本周延期，下周末再开 Gate（重做该 sprint 末端）
    - 选项 2：调整范围（去掉某功能，让总分够 🟢）
    - 选项 3：升级到 sponsor，需要追加资源
  - 任何 NO-GO 必须写 Lessons：什么时候应该提早发现

## G · 闸门后 24h（PM 必做）

1. **会后 1h 内**：发出会议记录（按 `gate-decision.md` 模板）+ 通知所有相关方
2. **会后 4h 内**：更新 `readiness_scorecard.md`，包括趋势图
3. **会后 24h 内**：发出业务方一页纸周报（按 `business-oneview.md` 模板）
4. **会后 48h 内**：召集相关 DRI 启动补救清单或下个 sprint

## H · 闸门反模式（PM 警惕）

| 反模式 | 危害 | PM 应对 |
|---|---|---|
| "差不多就先 GO 吧" | 把红灯变成黄灯然后变绿 = 自欺 | 坚持 9 项判据；记录在 Decision Log |
| sponsor 现场加项 | 已经定的 9 项被新维度污染 | "新维度作为下次闸门的第 10 项" |
| DRI 替别人打分 | 没有 ownership | 每维只由对应 DRI 念证据 |
| 会议拖到 90 min | 议程崩溃，决断质量降 | PM 60 min 强制散，未决项作为 ESC-NNN |
| 没有 NO-GO 历史 | 一直绿灯表明没有真正在审 | 至少需要 1 次 NO-GO，PM 才算合格 |

## I · Lessons Learned 模板

每次闸门后（无论 GO / CONDITIONAL / NO-GO）必须留 1 条 lessons：

```markdown
## Lessons from Gate W<n>

- 这次最危险的灯是哪个？为什么？
- 这个灯是被早发现的还是被晚发现的？
- 如果重来一次，PM 在哪一步就该早发现？
- 下次闸门要新增的"防早死"动作是什么？
```

归档到 `submission/governance/lessons_log.md`，累积到第 3 条时 PM 必须回顾一次"我作为 PM 是不是有系统性盲区"。

## 红线

- **闸门不补考** —— NO-GO 就 NO-GO，不允许会后给"补交作业"的特例
- **PM 不替 DRI 打分** —— 替了就是 PM 兜底，DRI 失去成长
- **业务方一页纸的总分 = 闸门评分卡总分** —— 不能两套数字
- **每场闸门必出 1 条 Lessons** —— 拒绝是把项目变更便宜的最大杠杆
