# Playbook 04 · 升级（escalation）处理

## A · 升级哲学

升级**不是抱怨**，升级是"我已经尝试了，现在需要 PM 的权限/资源/视野"。

> 一句话：**升级是项目最便宜的保险。**

> 团队的升级频率代表 PM 的"被信任度"。如果一个月没人升级 = PM 让团队不敢说 = PM 自己最大的盲区。

## B · 升级触发器（任一即报，不等 standup）

| # | 触发器 | 等级 |
|---|---|---|
| 1 | 任何 P0 任务预计 24h 内无法完成 | BLOCKER |
| 2 | 主干果场环境（生产后端 / CI / 签名仓库）损坏 | BLOCKER |
| 3 | 任何貌似安全/合规问题（口令泄露、PII 越界、插件越权）| BLOCKER |
| 4 | 同事间技术分歧 24h 内未能外部解决 | HIGH |
| 5 | 业主输入缺口将冲击下一个 W 节点 | HIGH |
| 6 | 第三方平台政策变更可能阻塞上架 | HIGH |
| 7 | 工具/账号缺位（CI 额度、Apple 账号）将影响 ≥3 天 | MEDIUM |
| 8 | 个人发现的潜在质量风险但未列入工单 | MEDIUM |

## C · PM 响应 SLA（不可商量）

| 等级 | 确认收到 | 给决断 | 关闭 |
|---|---|---|---|
| BLOCKER | 15 min | 2 h | 24 h |
| HIGH | 1 h | 24 h | 3 天 |
| MEDIUM | 4 h | 3 天 | 1 周 |

**PM 超时 = PM 失信用**。任何超时必须自我登记到月度 PM 复盘。

## D · 升级处理 SOP（4 步）

### Step 1 · 确认收到（SLA 内）

PM 用统一话术：

```
ESC-NNN 已收到，<等级>，<X 时间>前给决断。
当前并行让你做 <Y>，避免你卡死。
有进展我会主动同步。
```

**禁止**：
- "好的，我看看" ❌（没有时间承诺）
- "你这个不重要吧" ❌（先否认是 PM 大忌）
- "等周会讨论" ❌（违反 SLA）

### Step 2 · 五选一（PM 内部判断）

| 选项 | 适用 | 动作 |
|---|---|---|
| **A. 内部解决** | PM 有权拍板 | 写 D-NN 入 Decision Log，关闭升级 |
| **B. 转 RAID** | 短期无解，需持续监控 | 入 RAID Log，DRI 持续跟，定期复评 |
| **C. 上钻 sponsor** | 超 PM 权限 | 写 sponsor 一页纸，明确请求事项 |
| **D. 重排 sprint** | 需要释放/调换资源 | 在周会上做 sprint 范围调整 |
| **E. 召集临时评审** | 多方决策、争议大 | 24h 内开 ≤30min 临时会，输出 D-NN |

### Step 3 · 行动并书面同步

无论选哪条，PM 在 SLA 内给 DRI 一份**书面**：

```markdown
回 ESC-NNN ·

判决: <A/B/C/D/E>
理由: <2-3 句>
你接下来要做: <具体动作 + 时间>
我接下来要做: <具体动作 + 时间>
关联 D-NN: <如适用>
```

### Step 4 · 关闭升级

```markdown
## 关闭

- 关闭时间: 2026-MM-DD HH:MM
- 实际响应耗时: <分钟>
- 处置方案: <动作 + 入库位置>
- 后续监控: <是否进 RAID + DRI>
- Lessons: <1 句话能避免下次再发的话>
```

关闭后归档到 `submission/reports/escalations/ESC-NNN_*.md`，**不再修改**。

## E · 升级的"反内卷"红线

| 反模式 | 危害 | PM 应对 |
|---|---|---|
| 不报，加班自己扛 | 风险积累、burnout、PM 失去视野 | 鼓励早报，1on1 反复强调"升级=信任" |
| 报了又"算了" | 失去 SLA 历史数据 | 不允许撤回，可标 "RESOLVED_BY_REPORTER" 关闭 |
| 同事互推 | 找不到 DRI | PM 当场分配，"24h 内由 X 负责，无异议明早 9:30 前提" |
| PM 接到就否决 | 团队不再升级 | 即使最终不处理也要给"为什么不"的书面理由 |
| 用升级当作"逼宫" | 政治化 | PM 私聊指出，2 次后转 1on1 主题排查 |
| sponsor 直接收到（绕过 PM）| PM 失去信息流 | PM 立刻补承接，并不指责报告者 |

## F · 升级数据是 PM 的"经营仪表盘"

每月 PM 自查：

```markdown
| 月份 | BLOCKER | HIGH | MEDIUM | 平均关闭时长 | PM SLA 超时次数 |
|---|---|---|---|---|---|
| 2026-04 | 3 | 12 | 7 | 14h | 1 |
| 2026-05 | 1 | 8 | 11 | 8h | 0 |
```

**异常信号**：

- BLOCKER 数月持续上升 → 项目质量在恶化
- HIGH 数为 0 → 团队不敢报或 PM 不可信
- 平均关闭时长上升 → PM 决断力下降或被淹没
- SLA 超时 ≥ 2 次 → PM 必须自我反思工作量是否健康

## G · 7 个最常见升级场景的"PM 默认回应"

### G1 · "我下班前修不完这个 P0"
```
判决: 重排（D 选项）
原因: 没人 24h 工作
动作: 1) 你直接下班 2) 我把这个 P0 拆成 2 段 3) 明天 9:30 重新分工
保护机制: 写入 RAID R-NN 防止再现
```

### G2 · "A 和 D 在某个 API 设计上吵起来了"
```
判决: 临时评审（E 选项）
原因: 跨职能、24h 不解
动作: 1) 我今天 17:00 拉一个 30min 会 2) A 准备方案 X 利弊 3) D 准备方案 Y 利弊 4) 我拍板 + 写 D-NN
保护机制: 下次类似场景统一走"先 RFC 后实现"
```

### G3 · "业务方 5 天没回我隐私 URL 部署事"
```
判决: 上钻 sponsor（C 选项）
原因: 业主输入超 72h 是升级触发器
动作: 1) 我给 sponsor 发一页纸 2) 你继续按草稿版准备技术对接 3) 周五前不给我直接走"准 OK 版上线"
保护机制: 入 RAID D-NN
```

### G4 · "我发现 keystore 密码可能在 git history 里"
```
判决: 内部解决（A 选项）+ 转 RAID（B 选项）
原因: 安全 = BLOCKER
动作: 1) 我立即冻结所有 release tag 2) C 工号 1h 内启动密钥旋转 3) 写 D-NN 决定是否 revoke 当前 keystore
保护机制: 入 RAID R-NN + 复评 W4
```

### G5 · "Apple Developer 注册被退回"
```
判决: 重排 sprint（D 选项）
原因: 业主输入超期
动作: 1) 启动备用主体方案 2) iOS 路线推迟 2 周 3) 多出的人力转 Android 完善
保护机制: A-01 假设破裂，开 D-NN 重决"是否上 iOS"
```

### G6 · "C 工号说本周做不完 P0，因为请假 2 天"
```
判决: 重排（D 选项）
原因: 已发生客观条件变化
动作: 1) 不指责 2) PM 和 C 重排本周 3) PR 评审兜底由 PM 临时承担
保护机制: 检查是否有"单点失败"，启动"另一人能改"机制
```

### G7 · "Sponsor 提议加一个新功能"
```
判决: 内部解决（A 选项）—— 但要"温柔说不"
原因: 不在范围内，sprint 已锁
动作: 1) 我回 sponsor"加入 backlog，纳入下次闸门讨论" 2) 不动当前 sprint
保护机制: 入 Decision Log D-NN，下闸门拿出来评估
```

## 红线

- **不要去考验 SLA** —— SLA 是给团队的承诺，超一次失信
- **不要把升级当骂战记录** —— 升级是结构化求助，不是宣泄
- **不要替 sponsor 否决** —— sponsor 该看到的事 PM 必须如实传
- **不要让升级"消失"** —— 即使是误报也要正式关闭并归档
